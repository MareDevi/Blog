冒頭からの暴論：**C++ におけるムーブセマンティクス（Move Semantics）は、本質的に Rust における所有権の移動（Ownership Transfer）を手動でトリガーすることである。**

## Why we need it ?

C++11 以前は、オブジェクト（巨大な `std::vector` や `std::string` など）をある変数から別の変数へ渡したり、関数から返したりする場合、コンパイラは通常**ディープコピー（Deep Copy）**を実行していました。

- **コピーコンストラクタ：** 新しいメモリを割り当て -> データを複製 -> 古いオブジェクトを破棄。
- **無駄：** もし古いオブジェクトがすぐに破棄される運命にあるなら（例えば一時変数など）、「複製してから元を破棄する」というのは非常に愚かで高コストです。
  **ムーブセマンティクス**は、一時オブジェクトの複製を作る代わりに、そのリソース（ヒープメモリのポインタなど）を直接「盗む」ことを可能にします。

## 左辺値 vs 右辺値

ムーブセマンティクスを理解するには、まず左辺値と右辺値の違いを探る必要があります。

- **左辺値（Lvalue）**: 名前があり、アドレスがあり、永続的なオブジェクトです。
  つまり、左辺値に対してはアドレス取得 `&x` が可能です。
  例：`int a = 10`。ここで `a` は左辺値です。

- **右辺値（Rvalue）**: 名前がなく、アドレスがなく、消えゆくオブジェクト（つまり一時オブジェクト）です。
  右辺値は通常、リテラル、式の評価結果、または関数から返される非参照オブジェクトです。
  例：`10`、`x+y`、`func_return_obj()`。

> [!TIP] Tip
> 実は名前の通り、代入文 `int a = 10` において、
> 左側にあるのが左辺値、右側にあるのが右辺値です。

### 右辺値参照

C++11 では、新しい参照型である**右辺値参照**が導入されました。これは二重の `&` で表されます。

例：

```cpp
int& a       // 左辺値参照。左辺値にのみバインド可能
int&& a      // 右辺値参照。右辺値にのみバインド可能
```

```cpp
int a = 10;
int& ref1 = a;  // 合法：左辺値参照は左辺値にバインド
// int&& ref2 = a; // 非合法！a は左辺値なので、右辺値参照にはバインドできません

int&& ref3 = 10; // 合法：10 は右辺値
```

では、右辺値参照には何の意味があるのでしょうか？
右辺値参照は、コンパイラに対して次のように伝えています。「これはもうすぐ破棄される一時オブジェクトだ。好きに修正してもいいし、リソースを盗んでもいい。どうせ誰ももう使わないから。」

## ムーブセマンティクス

本題に戻りますが、ムーブセマンティクスとは何でしょうか？
ムーブセマンティクスとは、**右辺値参照**を利用してコンストラクタと代入演算子をオーバーロードし、いわゆる「リソースの窃盗」を実現することです。

### Example

ヒープメモリの管理を担当するクラス `Buffer` があるとします。

以前のコピーコンストラクタを使用する場合：

```cpp
// const Buffer& は、元のオブジェクトを変更せず、読み取るだけであることを意味します
Buffer(const Buffer& other) {
    // 1. 新しいメモリを割り当て
    data = new int[other.size];
    // 2. データを複製（遅い！）
    memcpy(data, other.data, other.size);
    size = other.size;
}
```

一方、ムーブセマンティクスを使用してムーブコンストラクタを実装する場合：

```cpp
// Buffer&& は、other が消えゆく値であり、変更可能であることを意味します
Buffer(Buffer&& other) noexcept {
    // 1. ポインタを直接盗む（激速！）
    data = other.data;
    size = other.size;

    // 2. 重要：元オブジェクトのポインタを空にする
    // そうしないと、other が破棄される時にこのメモリを delete してしまい、こちらもダングリングポインタになります（Double Free）
    other.data = nullptr;
    other.size = 0;
}
```

#### なぜ速いのか？

確かに `memcpy` は速いですが、フェラーリがいくら速くても瞬間移動には勝てません。
分解すると、主に 3 つの要因があります：

##### アルゴリズムの計算量（O(N) vs O(1)）

これが最も本質的な違いです。

- **Copy (memcpy ベース):** 1GB の `Buffer` オブジェクトがある場合、`memcpy` はバイトごとにソースアドレスから読み出し、宛先アドレスに書き込む必要があります。CPU は **10^9 バイト** を運搬しなければなりません。所要時間はデータ量に比例します（線形関係、**O(N)**）。
- **Move (ポインタ交換ベース):** `Buffer` が 1KB だろうが 100GB だろうが、ムーブコンストラクタがやることは 1 つだけ、**ポインタの代入**です。64 ビットシステムでは、ポインタは 8 バイトです。所要時間は定数で、ほぼ一瞬で完了します（**O(1)**）。

例えてみましょう：

- **Copy (`memcpy`)**: 本でいっぱいの家があります。引っ越しをします。すべての本を梱包し、新居に運び、開梱して並べます。動作は速い（SIMD）ですが、本が多ければ多いほど疲れます。
- **Move**: 新居の鍵を旧居の鍵と交換するだけです。本を動かす必要は全くありません。本はそこにあり、所有権が変わっただけです。

~~これが冒頭の暴論の由来です~~

##### 伴うメモリ割り当てコスト (System Call)

`memcpy` は単独で存在するわけではなく、通常その前には `new` があります。

```cpp
// Copy Constructor
data = new int[other.size];         // <--- 高価なシステムコール！
memcpy(data, other.data, size);     // <--- データの運搬
```

- **メモリ割り当て (`malloc`/`new`)**: これは重い操作です。OS は空きヒープブロックを探し、ヒープテーブルを更新し、同時実行ロックを処理し、ページフォールトさえ引き起こす可能性があります。
- **Move**: 新しいメモリを割り当てる必要はなく、既存のメモリブロックを直接引き継ぎます。

> [!TIP] Tip
> 小さなオブジェクトの場合、`new` のオーバーヘッドは `memcpy` よりも大きいことさえあります。

##### キャッシュの局所性 (Cache Locality)

- **Copy**: `memcpy` は大量のメモリ書き込みを伴います。真新しい（今 `new` したばかりの）大きなメモリ領域に書き込む際、そのメモリは CPU の L1/L2 キャッシュにない可能性が高いです。これは大量の **キャッシュミス** を引き起こし、CPU はメモリバスを待つことを余儀なくされ、パイプラインが停止します。
- **Move**: スタック上のいくつかのポインタ変数を修正するだけです。これらの変数は L1 キャッシュやレジスタに既にある可能性が非常に高いです。

### まとめ

はっきり言って、`memcpy` がダメなわけではなく、ディープコピー自体が極めて高価な操作なのです。

**対比：** コピーは「全く同じ家を建てる」こと。ムーブは「家の鍵を渡して、立ち去る」こと。

## `std::move` は一体何をしているのか？

これが最も誤解されやすい点です。**`std::move` は何も移動させません。**

これは単なる **強制型変換（キャスト）** です。左辺値（Lvalue）を強制的に右辺値参照（Rvalue Reference, `T&&`）にキャストします。

- **役割：** コンパイラに対してこう告げることです。「`x` は左辺値（名前がある）だけど、もう使わないことを保証するよ。右辺値（一時オブジェクト）として扱って、ムーブコンストラクタを呼び出していいよ！」

例：

```cpp
std::string a = "Hello World";
std::string b = a;            // コピーコンストラクタ呼び出し、a は依然有効
std::string c = std::move(a); // ムーブコンストラクタ呼び出し、a は空文字列になる（リソースは c に盗まれた）

// この時点で a にアクセスするのは安全ですが、未定義状態（通常は空）なので、その値に依存しないでください。
```
