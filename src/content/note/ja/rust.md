## イテレータとは何か

Rust において、イテレータは `Iterator` トレイトを実装した型です。その核心的な任務は、シーケンス内の項目を **オンデマンド（必要に応じて）** 生成することです。

**重要な特性：遅延 (Lazy)** は、Rust イテレータの最も重要な特性です。イテレータを作成するとき（例：`.iter()` を呼び出す）、あるいは関数をチェーンするとき（例：`.map()`）、**実際には何も起こりません**。「コンシューマ（消費者）」メソッド（`.collect()` や `for` ループでの使用など）を呼び出したときに初めて、イテレータは実際に動作を開始します。

もっと簡単に言えば、`Iterator` トレイトを実装するのに必要なメソッドは 1 つだけです：

```rust
pub trait Iterator {
    type Item;

    fn next(&mut self) -> Option<Self::Item>;
}
```

### イテレータの 3 つの形態

Rust の所有権システムの存在により、イテレータにも 3 つの存在形態があります：

| メソッド      | 取得タイプ | 説明                     | シナリオ                                                               |
| :------------ | :--------- | :----------------------- | :--------------------------------------------------------------------- |
| `iter()`      | `&T`       | 借用（読み取り専用参照） | データを読み取りたいが、変更したくないし、所有権も渡したくない場合。   |
| `iter_mut()`  | `&mut T`   | 可変借用（可変参照）     | トラバース中にコレクション内のデータを変更したい場合。                 |
| `into_iter()` | `T`        | 所有権を取得（値）       | トラバース後、元のコレクションは「消費」され、再利用できなくなります。 |

## コンシューマとアダプタ

イテレータを使用することは組立ラインを構築するようなもので、メソッドは **アダプタ (Adapters)** と **コンシューマ (Consumers)** の 2 種類に分類されます。

### アダプタ - 遅延変換者

これらのメソッドはイテレータを受け取り、新しいイテレータを返します。これらはすぐには実行されません！Shell で `cat file | grep text` と書くようなもので、Enter キーを押さなければ（つまりコンシューマがいなければ）、何も起こりません。

例：

- `map()`: 各要素を変換します。
- `filter()`: 要素をフィルタリングします。
- `take(n)`: 最初の n 個だけを取得します。
- `zip()`: 2 つのイテレータを「ジッパー」のように結合します。
- `enumerate()`: 要素にインデックス `(index, item)` を付加します。

### コンシューマ - 実行トリガー

これらのメソッドは `next()` を呼び出し、イテレータの実行を駆動し、最終結果を生成します。

例：

- `collect()`: 結果をコレクション（`Vec` や `HashMap` など）に収集します。
- `sum()`, `product()`: 数学演算。
- `find()`: 条件に合う要素を検索します。
- `for` ループ：本質的にはこれも一種のコンシューマです。

### Tip

上記を読めばわかるように、アダプタの後ろにコンシューマを加えなければ、イテレータ全体は実際には永遠に実行されません（もちろん、コンパイルしようとすると対応する警告が出ます）。

これが Rust イテレータの最も核心的な遅延評価（Lazy Evaluation）特性です。
アダプタ（Adapters、`map`、`filter` など）は水道管を敷設しているようなもので、コンシューマ（Consumers、`collect`、`for` など）は蛇口をひねるようなものだと想像してください。
どんなに複雑に水道管をつないでも（10 層のろ過、加熱、加圧を経たとしても）、最後に誰も蛇口をひねらなければ、水は管の中に流れ込みすらせず、最初の一滴さえ動きません。

#### なぜ `for` ループよりイテレータが良いのか？

1.  **パフォーマンス (Performance)**: Rust のコンパイラ (LLVM) はイテレータの最適化に極めて長けています。`filter` や `map` のようなチェーン呼び出しは通常、極めてコンパクトなアセンブリコードにコンパイルされ、コンパイラが境界チェックを排除できる（Bounds Check Elimination）ため、手書きの `for` ループより高速になることがよくあります。
2.  **安全性 (Safety)**: イテレータを使用することで、一般的な「境界外エラー」や「インデックスエラー」を回避できます。インデックス変数を手動で管理する必要が全くないからです。
3.  **可読性 (Readability)**: イテレータは「何をするか」（Map, Filter, Fold）を明確に表現しており、「どうやるか」（インデックス `i` を維持し、`i` を増やし、`i < len` をチェックする）ではありません。

## いくつかの一般的なメソッド

Rust の `Iterator` トレイトは汎用的なメソッド（`map`、`filter` など）を提供していますが、特定の機能は追加の状態や特別なロジックを必要とします。すべてのイテレータに重い負担をかけないために、Rust はオンデマンド拡張戦略を採用しています。特定のメソッドを呼び出すと、基本のイテレータをより高度な構造体にラップし、それによって新しい超能力を付与します。

### Peekable

標準の `next()` メソッドは「不可逆」です。一度見たら、データは「消費」（食べられ）されてしまいます。しかし、`peekable()` メソッドを呼び出すと、イテレータは `Peekable<T>` 型になり、内部に「覗き見た」データを格納するための「キャッシュスロット」が追加されます。

例：

```rust
fn main() {
    let nums = vec![1, 10, 20];
    // Peekable イテレータに変換
    // 注意：peek() は下層のイテレータからデータをプルしてキャッシュしようとする可能性があるため、mut が必要です
    let mut iter = nums.into_iter().peekable();

    // 1 回目の peek
    if let Some(&n) = iter.peek() {
        println!("覗き見：{}", n); // 出力 1
    }
    // もう一度 peek、データはまだある！
    println!("まだある：{:?}", iter.peek()); // Some(1)

    // 正式に消費
    println!("食べた：{:?}", iter.next()); // Some(1)

    // 次をもう一度 peek
    println!("次は：{:?}", iter.peek()); // Some(10)
}
```

### Enumerate

`enumerate()` メソッドはイテレータにインデックスを追加します。

例：

```rust
let tasks = vec!["宿題", "自習"];
// (usize, &str) になる
for (i, task) in tasks.iter().enumerate() {
    println!("{}: {}", i, task);
    // 出力：
    // 0: 宿題
    // 1: 自習
}
```

### Rev

`rev()` はイテレータを逆方向に出力します。

> 制限：`DoubleEndedIterator` を実装したイテレータでのみ使用可能です（例えば `Vec` は可能ですが、TCP ストリームからのイテレータは不可能です。ストリームの終わりを予知できないためです）。

```rust
let nums = vec![1, 2, 3];
for n in nums.iter().rev() {
    println!("{}", n); // 出力 3, 2, 1
}
```

### Cycle

無限ループ。必ず `take` や `break` と一緒に使用してください。

```rust
let pattern = vec!["A", "B"];
// 出力：A, B, A, B, A
for x in pattern.iter().cycle().take(5) {
    print!("{}, ", x);
}
```

### Fuse

ヒューズ（安全装置）。標準イテレータの規定では、一度 `next()` が `None` を返したら、理論上それ以降の呼び出しも `None` を返すべきですが、Rust はパフォーマンスのためにすべての実装がこれを守ることを強制していません。`fuse()` は強制的に保険をかけます。最初の `None` に遭遇したら、それ以降は永遠に `None` を返します。データソースの行儀が良いかどうかわからない複雑な低レベルロジックを書く場合に使用します。

## いくつかの高度な使用法？

### 1. 「内部状態」を持つ反復 (`scan` & `fold`)

通常の `map` はステートレス（Stateless）であり、現在の要素にしか関心がありません。しかし、**この要素を処理する際に前の要素の結果を知る必要がある**場合はどうすればよいでしょうか？

#### A. `scan`: 反復過程の蓄積器

`scan` は状態を持つ `map` のようなものです。可変の内部状態 `state` を維持し、反復ごとにそれを変更できます。

```rust
fn main() {
    let a = [1, 2, 3, 4];

    // state の初期値は 0
    // クロージャは (&mut state, item) を受け取る
    let running_totals: Vec<i32> = a.iter()
        .scan(0, |state, &x| {
            *state += x; // 内部状態を変更
            Some(*state) // 生成したい値を返す
            // None を返すと、反復はここで終了する（fuse のように）
        })
        .collect();

    println!("{:?}", running_totals);
    // 出力：[1, 3, 6, 10] (つまり 1, 1+2, 1+2+3...)
}
```

#### B. `fold`: 最終的な大統一

`collect` は実は `fold` の一種の特化です。`fold` はイテレータ内のすべてのものを単一の値に「折りたたみ」ます。

```rust
// 大量のログ行を 1 つの長い文字列にしたいと仮定します
let lines = vec!["Error: A", "Warning: B", "Info: C"];

let report = lines.iter().fold(String::from("Log Report:
"), |mut acc, &line| {
    acc.push_str(" - ");
    acc.push_str(line);
    acc.push_str("
");
    acc
});
// 結果は完全な String です。この書き方は String + String で何度もメモリ割り当てするよりはるかに効率的です
```

### 2. スライス (`windows` & `chunks`)

これらは実際にはスライス（slice）のメソッドですが、イテレータを返します。データストリーム処理、信号処理、テキスト分析に極めて有用です。

#### A. `windows(n)`: スライディングウィンドウ

重複するサブスライスを生成します。

```rust
let data = [10, 20, 15, 30, 40];

// ウィンドウサイズは 2、毎回右に 1 マス滑る
for slice in data.windows(2) {
    println!("Prev: {}, Curr: {}", slice[0], slice[1]);
}
// 出力：
// Prev: 10, Curr: 20
// Prev: 20, Curr: 15 ...
```

#### B. `chunks(n)`: バッチ処理

重複しないチャンク（塊）。

### 3. `by_ref`: ちょっと借りるだけ、持っていかない

これは非常に微妙ですが、極めて重要な高度なテクニックです。
イテレータに対して `take()` や `collect()` などのメソッドを呼び出すと、通常はそのイテレータ変数の所有権を **消費** してしまいます。もし **一部だけ消費** して、その後も同じイテレータを使い続けたい場合はどうすればよいでしょうか？

```rust
fn main() {
    let mut lines = vec!["Header1", "Header2", "---", "Body1", "Body2"].into_iter();

    // 1. "---" に出会うまでヘッダーを読み取る
    // 重要点：by_ref() を使用！
    // もし by_ref() を使わなければ、take_while が lines の所有権を持って行ってしまい、下で使えなくなります
    let headers: Vec<_> = lines.by_ref()
        .take_while(|line| *line != "---")
        .collect();

    println!("Headers: {:?}", headers);

    // 2. 同じ lines イテレータを使って残りの Body を読み続ける
    println!("Body start:");
    for line in lines {
        println!("{}", line); // 出力 Body1, Body2
    }
}
```

### 4. `iter::from_fn`: 虚空からイテレータを創造する

単純なカスタムイテレータを作るために、わざわざ新しい `struct` を書いて `Iterator trait` を実装する必要はありません。クロージャを使って直接作ることができます。

```rust
use std::iter;

fn main() {
    let mut count = 0;

    // next が Some を返す限り実行し続けるイテレータを作成
    let counter = iter::from_fn(move || {
        count += 1;
        if count < 5 {
            Some(count)
        } else {
            None
        }
    });

    for num in counter {
        println!("{}", num);
    }
}
```

### 5. 並列イテレータ (`Rayon`)

これはサードパーティライブラリ（`rayon`）に属しますが、Rust イテレータエコシステムの不可欠な一部と言っていいでしょう。

大量のデータを処理するイテレータチェーン（数百万行のログ処理など）がある場合、たった 2 行コードを修正するだけで、自動的に CPU の全コアを利用した並列処理が可能になります。

```rust
// Cargo.toml: rayon = "1.8"
use rayon::prelude::*; // 並列機能を導入

fn main() {
    let numbers: Vec<i64> = (0..1_000_000).collect();

    let sum: i64 = numbers
        .par_iter() // 注意：ここで iter() を par_iter() に変えました
        .map(|&x| x * 2) // この map は複数のスレッドで並列実行されます
        .sum();

    println!("{}", sum);
}
```

> [!TIP] 利点
>
> スレッド作成、ロック、メッセージパッシングのコードを一切書く必要がありません。Rust の所有権システムは、この並列処理が絶対的にデータ安全であることを保証します。
