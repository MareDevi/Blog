## Why we need it?

[Pull パターンと Push パターン](pull-push)の中で、Rust は Pull モデルに基づいていると述べました。つまり、Rust では、非同期タスク（Future）は自ら実行することはなく、エグゼキュータ（Executor）によって「プル（Pull）」されて初めて実行されます。

例えば、以下のような簡略化された `Future Trait` です：

```rust
trait Future {
    type Output;
    // ここに注意：self は Pin でラップされています
    fn poll(self: Pin<&mut Self>, cx: &mut Context) -> Poll<Self::Output>;
}
```

私たちが `async fn` を書くとき、Rust コンパイラは裏でこっそりとそれを **列挙型（Enum）** あるいは **構造体（Struct）**、いわゆる「ステートマシン」にコンパイルします。次回 `poll` で実行を再開したときにそれらを見つけられるように、関数内の **すべてのローカル変数** をパッケージ化して保存する必要があります。

次のようなコードを書いたと仮定します：

```rust
async fn my_task() {
    let mut buffer = [0u8; 1024];
    let reader = MyAsyncReader::new();

    reader.read(&mut buffer).await;
}
```

コンパイラはこれを以下のような構造体に変換します：

```rust
struct MyTaskState {
    buffer: [0u8; 1024],
    reader: MyAsyncReader,

    // フェーズ 1：read の完了待ち
}
```

`reader.read` が受け取るのは `buffer` の参照（`&mut buffer`）であることに注目してください。したがって、この状態では、`reader` は内部的に `buffer` を指すポインタを保持しています。

これこそが **自己参照（Self-Reference）** です。構造体の中のあるフィールドが、同じ構造体の中の別のフィールドを指している状態です。

さて、`Poll` モデルに戻りましょう。Rust の特徴は、デフォルトで **スタック割り当て** であり、**自由に移動（Move）できる** ことです。

制限を加えない場合、エグゼキュータは 2 回の `poll` の間に、この `MyTaskState` をメモリアドレス A からメモリアドレス B へ移動させる権利を完全に持っています（例えば、タスクキューを再配置するためや、`Box::new` でスタックからヒープへ移動させるためなど）。

では、もし移動が発生したらどうなるでしょうか？（元の構造体のメモリアドレスが `0x1000` にあると仮定します）

1.  構造体全体が新しいアドレス `0x2000` にコピーされます。
2.  `buffer` は現在 `0x2000` にあります。
3.  `reader` も `0x2000` にコピーされます。
4.  **しかし**、`reader` 内部のそのポインタは、元々 `0x1000`（古い buffer のアドレス）を格納していました。単純な `memcpy` はこのポインタを自動的に更新しません。
5.  結果：`reader` は現在 `0x1000` を指していますが、その場所はすでに回収されているか、上書きされているか、あるいはゴミデータです。
6.  **結果：** 次にそれを `poll` するとき、`reader` は `0x1000` にデータを書き込み、プログラムのクラッシュや深刻なデータ破損を引き起こします。

まさにこの衝突のため、Rust は `Pin` を導入しました。

## What is Pin ?

その名の通り、`Pin` は実際にはロックです。**自己参照でもある型（`!Unpin` を実装した型）にとって、一旦 `Pin` でラップされると、Safe Rust を介してその `&mut T` を取得することは永遠にできません。**

なぜ `&mut T` を取得できないのでしょうか？ `&mut T` を取得できて初めて、`std::mem::swap` や `std::mem::replace` を使って中の値を移動させることができるからです。

### Pin はどのように機能するか？

`Pin<P>` は、ポインタ `P`（`&mut T`、`Box<T>` など）をラップするラッパーです。その役割は、ラップされた値 `T` へのアクセス権限を **制限** することです：

- **大多数の型（`Unpin`）に対して:** `i32` や `String` のような通常の型は、移動させても安全です。これらは `Unpin` トレイトを実装しています。これらの型に対して、`Pin` は何の制限も設けず、自由に `&mut T` を取得して移動できます。
- **特殊な型（`!Unpin`）に対して:** `async` で生成された `Future` や自己参照構造体のように、これらは `Unpin` を実装していません。この種の型が `Pin` でラップされると（`Pin<&mut T>`）、Rust は生の `&mut T` を取得することを **禁止** します。
  - なぜ禁止するのか？ `&mut T` があれば、`std::mem::swap` や `std::mem::replace` を使って中の値を外に移動（Move out）できてしまうからです。
  - `Pin` はデータに檻をかけるようなものです。檻越しに操作（`poll` を呼び出す）することはできますが、それを取り出して場所を変えることはできません。

`Pin` のロジックは以下の通りです：

1.  Future を初めて `poll` するとき、まずそれを `Pin` する必要があります（例：`Box::pin(future)`）。
2.  これはコンパイラに対して「この Future はメモリ内に（スタック上であれヒープ上であれ）定住した」と伝えるのと同じです。
3.  これ以降、`Pin<&mut T>` を介してのみ操作できます。
4.  生の `&mut T` が手に入らないため、**物理的に** それを別の場所に移動させることはできません。
5.  動かない以上、その内部の自己参照ポインタ（`buffer` を指す `reader`）は永遠に安全で有効です。

## About C++

え？C++ の話もあるの？
はい。Rust のように標準ライブラリに存在するわけではありませんが、概念という形で現れています。そしてこれは、[C++ における右辺値参照とムーブセマンティクス](posts/c.md)で触れたムーブセマンティクスとちょうど関連しています。Cpp における Pin は、主に **ムーブセマンティクスの無効化** によって実現されます。

### 実装方法

C++ において、オブジェクトを「Pinned」（アドレス不変）にしたい場合、通常以下のような方法があります：

1.  **ムーブコンストラクタとムーブ代入演算子の削除：** これは最も明確な方法です。クラスが移動できない場合、コンパイラはそれをその場に留まる（あるいは、許可されているならコピーされる）ように強制します。
2.  **C++20 Coroutines（コルーチン）:** C++ 20 のコルーチンは Rust の Future と似ており、どちらも状態を保存する必要があります。C++ コンパイラは通常、コルーチンフレーム（Coroutine Frame）のために **ヒープ上** にメモリを割り当てます。ヒープ上で割り当てられるため、手動で delete しない限り、そのアドレスは固定されており、天然の「Pinned」となります。
3.  **ライブラリレベルの実装（Folly など）:** Facebook の Folly ライブラリなどには、`Pinned` と呼ばれる可能性のあるヘルパークラスがあったり、`std::unique_ptr` などのスマートポインタの特性（ヒープメモリを指し、オブジェクト自体は移動せず、ポインタの所有権だけが移動する）を利用してアドレスの固定を実現したりしています。

### Example:

```cpp
class PinnedClass {
public:
    PinnedClass() = default;
    // 移動を禁止
    PinnedClass(PinnedClass&&) = delete;
    PinnedClass& operator=(PinnedClass&&) = delete;

    // 通常はコピーにも注意が必要です。コピー後の自己参照ポインタはリダイレクトする必要があるかもしれないからです
    PinnedClass(const PinnedClass&) = delete;
    PinnedClass& operator=(const PinnedClass&) = delete;
};
```
