「Pull」および「Push」モデルは、コンピュータサイエンスにおける**データの流れ**と**制御権**に関する最も基本的な 2 つのアーキテクチャパターンです。

それらの本質的な違いは、**誰が「データ転送」というアクションを開始するか？**という点にあります。

- **Producer（生産者）:** データを生成する側。
- **Consumer（消費者）:** データを処理する側。

## Push モデル

**核心ロジック：** 生産者が能動的で、消費者は受動的です。生産者が新しいデータを生成すると、即座にデータを消費者に「プッシュ（押し出し）」します。消費者はデータがいつ来るかわからないため、常に受信できるように準備しておく必要があります。

- **生活の例え：** 携帯電話の SMS。いつ SMS が来るかわかりませんが、来ると電話が鳴ります（処理するように通知されます）。
- **コードでの表現：** コールバック関数（Callback）、イベントリスナー（Event Listener）、RxJS（Observable）。
- **メリット：**
  - **低遅延：** データは生成されるとすぐに配信されるため、リアルタイム性が非常に高いです。
  - **生産者の分離：** 生産者は送信するだけで、消費者の現在の状態（ブロックされていない限り）を気にしません。
- **デメリット：**
  - **消費者を圧倒しやすい（バックプレッシャー問題）:** 生産者がデータを生成する速度（例：毎秒 1000 件）が、消費者が処理する速度（例：毎秒 10 件）よりもはるかに速い場合、消費者は圧倒され、バッファオーバーフローやクラッシュを引き起こします。
  - **制御フローが複雑：** デバッグ時、スタックの追跡が困難になることがよくあります（コールバック地獄）。

## Pull モデル

**核心ロジック：** 消費者が能動的で、生産者は受動的です。消費者は自身の処理能力に応じて、生産者にデータを能動的に「リクエスト（要求）」します。消費者がリクエストしなければ、生産者は一時停止するか、データを一時保存します。

- **生活の例え：** ビュッフェ。あなた（消費者）はいつ皿を持って料理を取りに行くかを決め、食べ終わったら次の皿を取りに行きます。厨房（生産者）は料理を作ってそこに置いておくだけです。
- **コードでの表現：** イテレータ（Iterator/Generator）、Rust の `Future`、従来の HTTP リクエスト。

- **メリット：**
  - **自然なバックプレッシャー（Backpressure）サポート：** 消費者がレートを完全に制御します。処理しきれないときは、単に「プル（引く）」のをやめればよいだけで、システムはクラッシュしません。
  - **組み合わせが容易：** 「最初の 5 つを取る」、「フィルタリングする」などのロジックを簡単に実装できます（例：Rust の `Iterator` アダプタ）。
- **デメリット：**
  - **潜在的な遅延：** データが準備できていても、消費者が取りに来なければ、データは処理されません。
  - **ビジーウェイト（Busy Waiting）:** 消費者が「無限ループ」で絶えずプルし続ける（ポーリングする）と、空回りで CPU を浪費します。

## 非同期の比較：Rust vs C++

### Rust: Pull ベース

Rust において、`Future` は本質的に単なる **ステートマシン（状態機械）** のデータ構造です。

- **遅延（Laziness）:** `Future` を作成したとき（例えば `async` 関数を呼び出したとき）、**何も起こりません**。コードは 1 行も実行されません。それは単に「何をしたいか」を記述した構造体を生成しただけです。
- **Poll メカニズム：** この `Future` を **Executor（実行者）**（`tokio` や `async-std` など）に渡すか、別の `async` ブロック内で `.await` したときに初めて、Executor はその `poll()` メソッドを呼び出します。
- **ワークフロー:**
  1.  Executor が `future.poll()` を呼び出します。
  2.  Future は実行を試みます。ブロックに遭遇した場合（ソケットデータ待ちなど）、OS/Reactor に `Waker`（ウェーカー）を登録し、`Poll::Pending`（まだ準備できていない）を返します。
  3.  Executor は `Pending` を受け取ると、別のタスクを処理しに行きます（つまり、CPU はこの時点では待ちません）。
  4.  **重要点：** データが到着すると、OS は Reactor に通知し、Reactor は以前に登録された `Waker.wake()` を呼び出します。
  5.  `wake()` は直接コードを実行するのではなく、Executor にこう伝えます。「おい、このタスクは準備できたかもしれないぞ。もう一度 **プル（Poll）** してみろ。」
  6.  Executor は再び `poll()` を呼び出します。今回 Future は `Poll::Ready(result)` を返し、タスクは完了します。

> [!TIP] 比喩
> あなた（Executor）は上司、Future は従業員です。
> あなたは能動的に従業員に「仕事は終わったか？」（Poll）と聞かなければなりません。
> 従業員が終わっていなければ、あなたの電話番号を控えます（Waker）。
> 外部条件が満たされると（書類が届いたなど）、従業員はあなたに電話します。「上司、もう一度聞きに来ていいですよ。」
> あなたがもう一度聞きに行って（Poll）初めて、従業員は結果を渡します。
> もしあなたが聞かなければ、従業員は永遠にそこに座ったまま動きません。

### C++20: Push

C++20 のコルーチン標準自体は、多様なパターンを可能にする構築メカニズム（`promise_type`、`awaitable`）を提供していますが、主流の実装や一般的な理解では、**Eager（性急な）** 実行と **Push** 再開の傾向があります。

- **性急な実行（Eager）:** C++ では、コルーチン関数を呼び出すと、通常すぐに実行が開始され、最初の中断点（suspension point, `co_await`）に遭遇するまで走ります。
- **コールバック/再開メカニズム：** コルーチンが `co_await` で I/O 待ちになると、中断（Suspend）し、コールバックを登録します（通常 `std::coroutine_handle` を介して）。
- **ワークフロー:**
  1.  コルーチンが実行され、非同期操作（ファイル読み込みなど）を開始します。
  2.  下位の I/O ライブラリがリクエストを引き継ぎ、コルーチンは中断します。
  3.  OS が I/O 操作を完了すると、コールバックをトリガーします。
  4.  **重要点：** このコールバックは `handle.resume()` を直接呼び出し、前回停止した場所からコルーチンを **プッシュ（Push）** して実行を継続させます。

> [!TIP] 比喩
> あなた（メインスレッド）はレストランへ注文しに行きます（コルーチン起動）。
> ウェイター（ランタイム）は呼び出しベルを渡します。
> 料理ができると（I/O 完了）、ベルが振動するか、あるいはウェイターが直接料理をテーブルに運び（Resume）、あなたに食事を始めるよう促します（Push）。
> あなたは数分おきに厨房へ「料理できた？」と聞きに行く必要はありません。

### 両者の比較

| **特性**                     | **Rust (Pull / Poll)**                                                                                                                                               | **C++ (Push / Callback-style)**                                                                                                                                   |
| :--------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **起動方式**                 | **Lazy（遅延）**。関数呼び出しは Future を返すだけで、`.await` または `spawn` しないと実行されない。                                                                 | **Eager（性急）**。関数呼び出しは通常、最初の `co_await` まで即座に実行を開始する。                                                                               |
| **ステートマシンの場所**     | **インライン/スタック（通常）**。コンパイラが生成するステートマシンは構造体であり、他の Future にネストでき、最終的に 1 つの大きなステートマシンにコンパイルされる。 | **ヒープ割り当て**。コルーチンは通常、コルーチンフレームを保存するためにヒープメモリを割り当てる必要がある（Halo 最適化で一部削除可能）。                         |
| **スケジューリングロジック** | **ステートマシン駆動**。Executor が繰り返し `poll` を呼び出す。状態を明示的に進める必要がある。                                                                      | **イベント駆動**。完了イベントが直接 `resume` をトリガーする。                                                                                                    |
| **タスクキャンセル**         | **極めて簡単**。Future を直接 `Drop` すればよい。`poll` しなければ動作は止まる。                                                                                     | **非常に困難**。コルーチンは既に実行中かコールバックチェーンに保持されている可能性があるため、停止を通知するための明示的な `CancellationToken` メカニズムが必要。 |
| **メモリオーバーヘッド**     | **極めて低い**。待機中の各タスクに独立したヒープ領域を割り当てる必要がないため（ゼロコスト抽象化）。                                                                 | **高め**。各並行タスクは通常、独立したヒープ割り当てフレームを必要とする（コンパイラが積極的に最適化しない限り）。                                                |
