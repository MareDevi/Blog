## Why we need it?

In [Pull and Push Patterns](pull-push), we mentioned that Rust is based on the Pull model. That is to say, in Rust, an asynchronous task (Future) will not run by itself; it must be Pulled by an executor to execute.

For example, this simplified `Future Trait`:

```rust
trait Future {
    type Output;
    // Note here: self is wrapped by Pin
    fn poll(self: Pin<&mut Self>, cx: &mut Context) -> Poll<Self::Output>;
}
```

When we write an `async fn`, the Rust compiler secretly compiles it into an **Enum** or **Struct** behind the scenes, which is what we often call a "State Machine". It must package and store **all local variables** in the function so that they can be found when `poll` resumes execution next time.

Suppose we write a piece of code like this:

```rust
async fn my_task() {
    let mut buffer = [0u8; 1024];
    let reader = MyAsyncReader::new();

    reader.read(&mut buffer).await;
}
```

The compiler will convert it into a struct similar to the following:

```rust
struct MyTaskState {
    buffer: [0u8; 1024],
    reader: MyAsyncReader,

    // Phase 1: Waiting for read to complete
}
```

We can notice that `reader.read` accepts a reference to `buffer` (`&mut buffer`), so in this state, `reader` internally holds a pointer pointing to `buffer`.

And this is **Self-Reference**: a field in a struct points to another field in the same struct.

Now let's go back to the `Poll` model. Rust is characterized by default **stack allocation** and **arbitrary movement (Move)**.

Without restrictions, the executor has the full right to move this `MyTaskState` from memory address A to memory address B between two `poll`s (for example, to rearrange the task queue, or move from stack to heap via `Box::new`).

So what are the consequences if a move occurs? (Let's assume the original memory address of the struct is at `0x1000`)

1.  The entire struct is copied to the new address `0x2000`.
2.  `buffer` is now located at `0x2000`.
3.  `reader` is also copied to `0x2000`.
4.  **However**, the pointer inside `reader` originally stored `0x1000` (the address of the old buffer). A simple `memcpy` will not automatically update this pointer.
5.  Result: `reader` now points to `0x1000`, and that place may have been reclaimed, overwritten, or contain garbage data.
6.  **Consequence:** Next time you `poll` it, `reader` writes data to `0x1000`, causing a program crash or severe data corruption.

It is because of this conflict that Rust introduced `Pin`.

## What is Pin?

Just like its name, `Pin` is actually a lock: **For those types that are also self-referential (types that implement `!Unpin`), once wrapped by `Pin`, you can never get its `&mut T` through Safe Rust.**

Why can't we get `&mut T`? Because only if you have `&mut T`, you can use `std::mem::swap` or `std::mem::replace` to move the value inside out.

### How does Pin work?

`Pin<P>` is a wrapper around a pointer `P` (like `&mut T`, `Box<T>`). Its function is to **restrict** access permissions to the wrapped value `T`:

- **For most types (`Unpin`):** Ordinary types like `i32`, `String` are safe to move. They implement the `Unpin` trait. For these types, `Pin` has no restrictions, and you can freely get `&mut T` and move it.
- **For special types (`!Unpin`):** Like `Future` generated by `async` or self-referential structs, they do not implement `Unpin`. When this type is wrapped by `Pin` (`Pin<&mut T>`), Rust **forbids** you from getting the raw `&mut T`.
  - Why forbid? Because with `&mut T`, you can use `std::mem::swap` or `std::mem::replace` to move the value out (Move out).
  - `Pin` is like putting a cage around the data. You can operate on it through the cage (call `poll`), but you can't take it out and change its position.

The logic of `Pin` is:

1.  When you `poll` a Future for the first time, you must `Pin` it first (e.g., `Box::pin(future)`).
2.  This is equivalent to telling the compiler: "This Future has settled down in memory (whether on the stack or heap)."
3.  From then on, you can only operate on it through `Pin<&mut T>`.
4.  Because you can't get the raw `&mut T`, you are **physically unable** to move it elsewhere.
5.  Since it won't move, its internal self-referential pointer (`reader` pointing to `buffer`) will always be safe and valid.

## About C++

What? C++ is involved too?
Yes, although it doesn't exist in the standard library like Rust, it appears in the form of a concept. And this happens to be related to the move semantics we mentioned in [Rvalue References and Move Semantics in C++](posts/c.md): Pin in Cpp is mainly implemented by **disabling move semantics**.

### Implementation

In C++, if you want an object to be "Pinned" (address immutable), common practices are:

1.  **Delete move constructor and move assignment operator:** This is the most explicit way. If a class cannot be moved, the compiler will force it to stay in place (or be copied, if you allow it).
2.  **C++20 Coroutines:** C++20 coroutines are similar to Rust's Future and both need to save state. The C++ compiler usually allocates memory for the Coroutine Frame on the **heap**. Because it is allocated on the heap, as long as you don't manually delete it, its address is fixed and naturally "Pinned".
3.  **Library-level implementation (like Folly):** Libraries like Facebook's Folly have helper classes possibly named `Pinned`, or use features of smart pointers like `std::unique_ptr` (pointing to heap memory, the object itself doesn't move, only the ownership of the pointer moves) to achieve address fixing.

### Example:

```cpp
class PinnedClass {
public:
    PinnedClass() = default;
    // Disable move
    PinnedClass(PinnedClass&&) = delete;
    PinnedClass& operator=(PinnedClass&&) = delete;

    // Usually also be careful with copy, because the self-referential pointer after copy might need redirection
    PinnedClass(const PinnedClass&) = delete;
    PinnedClass& operator=(const PinnedClass&) = delete;
};
```
